1. Resumen Ejecutivo

NovoTechFlow es una aplicación web diseñada específicamente para NOVOTECHNO DE COLOMBIA, un canal de cómputo empresarial que comercializa laptops, servidores, workstations, software, servicios y accesorios de marcas como Dell, Lenovo y HP. La herramienta permitirá a los equipos comerciales crear, versionar y exportar propuestas comerciales profesionales en formato PDF, con cálculo automático de márgenes, costos y precios de venta.

El sistema incluye una aplicación web principal y un aplicativo de escritorio (Tauri + Rust) para sincronización de archivos locales con el servidor.

Objetivos principales

-   Centralizar la creación de propuestas comerciales en una sola plataforma.

-   Automatizar el cálculo de márgenes, impuestos (IVA) y precios de venta.

-   Generar PDFs profesionales con branding corporativo consistente.

-   Controlar versiones de propuestas de forma inmutable.

-   Sincronizar archivos locales con el servidor mediante la app de escritorio.

-   Dashboard de seguimiento con exportación de informes a Excel.

2. Arquitectura del Sistema

El sistema sigue una arquitectura cliente-servidor con monorepo, separando frontend, backend y aplicativo de escritorio en paquetes independientes gestionados por Turborepo.

2.1 Diagrama de alto nivel

[Navegador Web] ↔ [React + Vite (SPA)] ↔ [NestJS API REST] ↔ [PostgreSQL]

[Tauri Desktop App] ↔ [NestJS API REST] ↔ [File Storage / S3]

2.2 Componentes principales

-   Frontend Web (React 18 + Vite): SPA con enrutamiento protegido por roles. Comunicación vía REST API con el backend. UI responsiva con Tailwind CSS.

-   Backend API (NestJS 10): API RESTful con autenticación JWT, controladores por módulo (auth, users, proposals, products, pdf-generation, file-sync). Validación con class-validator y class-transformer.

-   Base de Datos (PostgreSQL 15 + Prisma ORM): Modelo relacional normalizado. Migraciones versionadas con Prisma Migrate.

-   Aplicativo de Escritorio (Tauri + Rust): App nativa liviana para sincronización bidireccional de archivos. Login con credenciales del sistema web.

5. Módulos y Pantallas de la Aplicación

5.1 Pantalla 1 — Login

Pantalla de inicio de sesión unificada para todos los perfiles de usuario.

Funcionalidades:

-   Campos: correo electrónico y contraseña.

-   Autenticación JWT con tokens de acceso y refresh token.

-   Redirección automática según rol: Admin ve panel completo, Comercial ve dashboard.

-   Branding NovoTechFlow con paleta corporativa.

Especificaciones técnicas:

-   Endpoint: POST /api/auth/login → devuelve { accessToken, refreshToken, user }.

-   Guards de NestJS: AuthGuard (JWT) y RolesGuard (ADMIN/COMMERCIAL).

-   Frontend: React Router con rutas protegidas (PrivateRoute component).

5.2 Pantalla 2 — Panel de Administración (solo Admin)

Panel exclusivo para usuarios con rol ADMIN. Contiene dos módulos principales: gestión de usuarios y acceso al generador de propuestas.

Módulo 2.1 — Gestión de Usuarios:

-   Formulario de creación: nombre, correo, contraseña, rol (Admin/Comercial), nomenclatura.

-   Validación de nomenclatura única: se genera automáticamente de las iniciales del nombre (ej: Luis Maldonado → LM) con posibilidad de edición.

-   Vista previa del formato de propuesta: COT-[NOMENCLATURA]####. Ejemplo: COT-LM0001.

-   Tabla de usuarios existentes con acciones: editar, activar/desactivar, resetear contraseña.

-   Endpoints: GET/POST/PATCH /api/users (protegidos con RolesGuard ADMIN).

Módulo 2.2 — Acceso a Propuestas:

-   Acceso directo al dashboard de propuestas (Pantalla 3).

-   El admin puede ver todas las propuestas de todos los comerciales.

5.3 Pantalla 3 — Dashboard de Propuestas

Panel principal que muestra el resumen consolidado de todas las propuestas comerciales.

Componentes del Dashboard:

-   Tarjetas de resumen (KPIs): total propuestas, valor total cotizado, propuestas activas, propuestas completadas, margen promedio.

-   Tabla de propuestas: código (COT-XX####), cliente, asunto, fecha emisión, fecha actualización, estado (borrador/completada/archivada), versión actual, valor total.

-   Filtros avanzados: por comercial, por cliente, por rango de fechas, por estado, por rango de valor.

-   Botón de exportación a Excel: genera archivo .xlsx con los datos filtrados.

-   Acciones rápidas por fila: ver, editar (si está en borrador), clonar, descargar PDF, crear nueva versión.

Especificaciones técnicas:

-   Endpoint: GET /api/proposals?filters=... con paginación, ordenamiento y filtros.

-   Endpoint export: GET /api/proposals/export-excel con los mismos filtros.

-   Frontend: React Table (TanStack Table) con virtualización, Recharts para gráficos.

5.4 Pantalla 4 — Constructor de Propuesta Comercial

Pantalla principal de creación y edición del contenido de la propuesta. Aquí se definen los artículos, secciones, escenarios y el contenido que conformará el PDF final.

Encabezado de la propuesta:

-   Código autogenerado (COT-[NOMENCLATURA]####).

-   Nombre del cliente (campo de texto con autocompletado de clientes previos).

-   Asunto de la propuesta.

-   Fecha de emisión, validez en días y fecha de vigencia.

Configurador de artículos:

-   Campos seleccionables: tipo (producto/software/servicio), marca, nombre, part number, descripción técnica.

-   Cuadros de texto enriquecido para características detalladas.

-   Acciones por artículo: clonar, editar, eliminar, reordenar (drag & drop).

Botones flotantes (Floating Action Buttons):

-   Agregar Sección: añade una nueva sección de contenido.

-   Agregar Escenario: crea un escenario alternativo de cotización.

-   Crear Versión: genera una nueva versión de la propuesta.

-   Agregar Página: inserta páginas de texto, imágenes o tablas.

Secciones de contenido para el PDF:

-   Páginas de texto: editor rich text (TipTap/Slate) con formato básico.

-   Páginas de imágenes: upload de imágenes con posicionamiento y título.

-   Páginas de tablas: tablas personalizables con datos editables.

-   Descripción técnica: autogenerada desde los artículos configurados.

5.5 Pantalla 5 — Calculadora de Costos y Márgenes

Tabla dinámica interconectada con los artículos creados en la Pantalla 4. Todos los cálculos se actualizan en tiempo real.

Funcionalidades:

-   Tabla editable en línea: cada fila es un artículo con columnas para costo, margen %, precio unitario, cantidad, subtotal, IVA, total.

-   Cálculo automático de margen: Precio = Costo / (1 - Margen%). Al modificar uno se recalculan los demás.

-   Selector de escenarios: vista por escenario con totales independientes.

-   Indicador de productos gravados (IVA 19%) vs no gravados.

Totales calculados en tiempo real:

-   Subtotal antes de impuestos (gravados + no gravados).

-   Base gravable e IVA (19%).

-   Total IVA incluido.

-   Margen total de la propuesta (promedio ponderado).

-   Costo total vs precio total (resumen de rentabilidad).

Restricciones:

-   Solo se pueden agregar artículos previamente creados en la Pantalla 4.

-   Los cambios de precio/margen en esta pantalla se reflejan en el PDF de venta.

Especificaciones técnicas:

-   Frontend: estado centralizado con Zustand o React Context. Cálculos en cliente con debounce para autosave.

-   Backend: validación de cálculos en servidor antes de guardar (doble verificación).

6. Aplicativo de Escritorio (Tauri + Rust)

Aplicación nativa liviana construida con Tauri y Rust, con interfaz web embebida. Su única función es la sincronización bidireccional de archivos con el servidor de la aplicación web.

6.1 Funcionalidades

-   Login: inicio de sesión con las mismas credenciales de la app web (JWT compartido).

-   Creación automática de estructura de carpetas: Propuestas/{COT-XX####}/{Versión}/{Archivos}.

-   Sincronización ascendente: sube archivos locales nuevos/modificados al servidor.

-   Sincronización descendente: descarga archivos del servidor al equipo local.

-   Verificación de integridad: comparación por checksum SHA-256.

-   Indicador de estado: sincronizado, pendiente, error.

6.2 Arquitectura técnica

-   Core en Rust: manejo de filesystem, cálculo de hashes, upload/download HTTP.

-   Frontend embebido: React (compartido del monorepo) con vista mínima.

-   Comunicación: Tauri Commands (invoke) para llamar funciones Rust desde JS.

-   Endpoints del backend: POST /api/files/upload, GET /api/files/download/:id, GET /api/files/sync-status.

7. Generación de PDF

El sistema genera documentos PDF profesionales a partir del contenido configurado en la propuesta. El administrador puede diseñar y organizar las páginas plantilla que estarán disponibles para todos los usuarios.

7.1 Estructura del PDF

-   Portada: branding corporativo, nombre del cliente, asunto, fecha, código de propuesta.

-   Páginas de introducción: plantilla configurable por el administrador.

-   Páginas de contenido: texto, imágenes y tablas configuradas en Pantalla 4.

-   Descripción técnica: ficha de cada artículo con especificaciones detalladas.

-   Tabla de precios por escenario: productos, cantidades, precios unitarios, subtotales, separación gravados/no gravados, base gravable, IVA 19%, total IVA incluido.

-   Términos y condiciones: plantilla estándar configurable.

7.2 Motor de generación

Se recomienda un enfoque híbrido: React PDF (@react-pdf/renderer) para templates programables en JSX, con Puppeteer como fallback para rendering complejo con CSS avanzado. El backend genera el PDF y lo almacena vinculado a la versión de la propuesta.

7.3 Administración de plantillas PDF

-   El administrador accede a un editor de plantillas donde puede crear, editar y ordenar las páginas estándar.

-   Tipos de plantilla: portada, introducción, términos y condiciones, páginas personalizadas.

-   Las plantillas se almacenan como JSONB y se renderizan como componentes React al generar el PDF.

8. Sistema de Versionado de Propuestas

El versionado es un pilar fundamental de NovoTechFlow. Garantiza trazabilidad y previene modificaciones accidentales a propuestas ya entregadas.

8.1 Reglas de versionado

-   Una propuesta en estado DRAFT puede editarse libremente.

-   Al completar una propuesta (estado COMPLETED), se genera un snapshot JSONB inmutable de todo el contenido y se bloquea (is_locked = true).

-   Para modificar una propuesta completada, el usuario debe crear una nueva versión. Esto clona el contenido actual como base editable.

-   Cada versión tiene su propio PDF asociado.

-   Alternativa: crear una propuesta completamente nueva (nuevo código COT).

8.2 Flujo de estados

DRAFT ──[completar]──▶ COMPLETED (locked) ──[nueva versión]──▶ DRAFT (v2)

COMPLETED ──[archivar]──▶ ARCHIVED

13. Despliegue e Infraestructura

Opción recomendada: VPS / Cloud

-   Servidor: VPS Linux (Ubuntu 22.04 LTS) con Docker Compose.

-   PostgreSQL 15 en contenedor Docker con volúmenes persistentes.

-   Backend NestJS: contenedor Docker con Node.js LTS.

-   Frontend React: build estático servido por Nginx o Caddy.

-   Almacenamiento de archivos: S3-compatible (MinIO self-hosted o AWS S3).

-   SSL/TLS: Let’s Encrypt con Caddy o Certbot.

-   CI/CD: GitHub Actions para build, test y deploy automático.

-   Backups automáticos: pg_dump diario + copias S3.

14. Riesgos y Mitigación

  ---------------------------- ----------- ----------------------------------------------------------------------------------------------------------
  Riesgo                       Impacto     Mitigación

  Complejidad del PDF          Alto        Prototipar la generación PDF desde la Fase 1 con un template básico. Iterar incrementalmente.

  Rendimiento tabla dinámica   Medio       Virtualizar la tabla con TanStack Virtual. Debounce en cálculos. Optimistic updates.

  Sincronización de archivos   Medio       Implementar checksums SHA-256, reintentos automáticos, cola de sincronización.

  Migraciones de BD            Medio       Usar Prisma Migrate con revisión en PR. Backups antes de cada migración.

  Concurrencia de edición      Bajo        Bloqueo optimista con campo updated_at. Notificación si otro usuario modificó.

  Compatibilidad Windows       Medio       Probar Tauri en múltiples versiones de Windows. Incluir Visual Studio C++ Build Tools como prerequisito.
  ---------------------------- ----------- ----------------------------------------------------------------------------------------------------------

16. Estándares de Codificación y Mejores Prácticas (Senior Level)

Para asegurar la escalabilidad, mantenibilidad y calidad técnica de NovoTechFlow, todo el código debe cumplir con:

16.1 Principios SOLID y Arquitectura
- Responsabilidad Única (SRP): Cada clase, módulo o función debe tener una única razón para cambiar.
- Abierto/Cerrado (OCP): El sistema debe permitir extender funcionalidades sin modificar el código existente.
- Segregación de Interfaces e Inyección de Dependencias: Uso extensivo de DI para desacoplar lógica de infraestructura.

16.2 Diseño y Limpieza (Clean Code)
- Legibilidad: Nombres de variables, funciones y clases descriptivos (evitar siglas o nombres genéricos).
- Early Returns: Evitar anidamientos profundos de estructuras 'if/else' para mejorar la claridad del flujo.
- Comentarios de Contexto: Documentar el "por qué" de decisiones técnicas complejas, dejando que el código explique el "qué".

16.3 Documentación Profesional
- Backend (NestJS/TS): Uso obligatorio de TSDoc para todos los métodos, controladores y servicios, especificando parámetros, tipos y excepciones.
- Frontend (React): Documentación de Props, Componentes de alto nivel y Hooks personalizados.

16.4 Arquitectura de Software
- Separación de Responsabilidades: Clara distinción entre controladores (infraestructura), servicios (lógica de negocio) y entidades/repositorios (acceso a datos).
